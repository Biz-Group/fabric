## Phase 6: Post-Call Pipeline (Convex Functions)

- [ ] **Create `fetchConversation` Convex action**
  - **Trigger**: Called by the frontend via `useMutation`/`useAction` after `onDisconnect` fires (polling path â€” recommended for POC)
  - **Input**: `elevenlabs_conversation_id` and `processId` and `contributor_name` from the frontend
  - **Logic**:
    1. Poll `GET https://api.elevenlabs.io/v1/convai/conversations/{conversation_id}` with `xi-api-key` header
    2. Poll every ~2 seconds until the response `status` field = `done`
    3. Implement max-retry counter: 30 retries Ă— 2s = 60s timeout
    4. If max retries exceeded and still `processing`, use `ctx.runMutation` to insert a record with `status: 'processing'` so the frontend can detect it via Convex reactivity
    5. On `status: 'done'`, extract from the response:
       - `transcript` â†’ store in `conversations.transcript` (list of message objects with role, content, `time_in_call_secs`)
       - `analysis.transcript_summary` â†’ store in `conversations.summary` (this is the ElevenLabs-generated summary, **no Claude call needed**)
       - `analysis` (full object including success evaluation + data collection results) â†’ store in `conversations.analysis`
       - `metadata.call_duration_secs` â†’ store in `conversations.duration_seconds`
    6. Use `ctx.runMutation` to insert a new row into `conversations` table with all extracted fields + `processId`, `contributor_name`, `elevenlabs_conversation_id`, `status: 'done'`
    7. After successful insert, call `regenerateProcessSummary` (via `ctx.runAction` or `ctx.scheduler.runAfter`)
    8. On `status: 'failed'` from ElevenLabs, insert record with `status: 'failed'`
  - **Security**: `xi-api-key` stored in Convex environment variables (set via `npx convex env set`) â€” never exposed to the client
  - **Error handling**: Throw appropriate errors; log for debugging

- [ ] **Create `regenerateProcessSummary` Convex action**
  - **Trigger**: Called after `fetchConversation` successfully inserts a conversation
  - **Input**: `processId`
  - **Logic**:
    1. Use `ctx.runQuery` to fetch ALL conversation summaries for the given `processId` from the `conversations` table (where `status = 'done'`, ordered by `_creationTime`)
    2. Construct a prompt for Claude Sonnet (via OpenRouter) with the synthesis instruction from PRD section 3.4:
       > "You are synthesizing multiple employee accounts of a single business process. Combine these into a coherent narrative that describes the full process end-to-end, noting which contributors handle which parts, and highlighting any overlaps or gaps."
    3. Include all conversation summaries as input context (these are short summary strings, not full transcripts â€” lightweight call)
    4. Call OpenRouter API (`OPENROUTER_API_KEY` from Convex environment variables, model: `anthropic/claude-sonnet-4`) using the OpenAI-compatible endpoint (`https://openrouter.ai/api/v1/chat/completions`) to generate the rolling summary
    5. Use `ctx.runMutation` to update `processes.rolling_summary` with Claude's response for the given `processId`
  - **Cost note**: This is the **only LLM cost** in the system â€” ElevenLabs handles all conversation-level summarization natively
  - **Concurrency note (PRD section 8.1)**: For POC, accept last-write-wins if two summaries fire near-simultaneously. The second call will include both conversation summaries anyway. Production would add a debounce mechanism.

- [ ] **Wire the complete post-call data flow (end-to-end)**
  - Implement the full sequence from PRD section 3.5 "Data flow (POC â€” polling path)":
    1. User starts session â†’ `conversation.startSession()` â†’ `onConnect` stores `conversationId`
    2. User ends session â†’ `onDisconnect` fires with `reason`
    3. Frontend calls `fetchConversation` Convex action with `{ elevenlabs_conversation_id, processId, contributor_name }`
    4. Convex action polls ElevenLabs API until `status = done`
    5. Convex action extracts transcript + summary (from `analysis`) + data collection results â†’ inserts into `conversations` table via `ctx.runMutation`
    6. Convex action calls `regenerateProcessSummary` â†’ Claude synthesizes all summaries â†’ updates `processes.rolling_summary`
    7. Convex reactivity auto-updates the frontend â†’ UI refreshes with new conversation + updated summary (no manual subscriptions needed)
  - Handle the case where the Convex action times out (insert `processing` record, frontend detects via reactive `useQuery`)

- [ ] **Build the post-call loading state**
  - After `onDisconnect` with `reason: "user"` or `"agent"`, transition the recording modal to a loading/processing state:
    - Show **ShimmeringText** component: "Processing your conversation..."

- [ ] **Create `getAudio` Convex HTTP action (audio proxy)**

  - **Endpoint pattern**: Convex HTTP action registered via `httpAction` (e.g., `GET /audio/{elevenlabs_conversation_id}`)
  - **Logic**:
    1. Receive `elevenlabs_conversation_id` as a URL parameter
    2. Call `GET https://api.elevenlabs.io/v1/convai/conversations/{conversation_id}/audio` with `xi-api-key` header
    3. **Stream** the raw audio response (MP3) directly back to the client â€” do not buffer the entire file

  - **Security**: The `xi-api-key` is added server-side only â€” the frontend never touches it. The `agentId` can be public, but the API key must not be exposed client-side (PRD section 8.1)
  - **No storage**: Fabric does NOT store audio files. Audio is served on-demand from ElevenLabs. The URL is deterministic from `elevenlabs_conversation_id` (no `audio_url` column in the database). No additional ElevenLabs credits consumed â€” retrieval is a read operation (PRD section 3.3.5)
  - **Tradeoff acknowledged**: Playback depends on ElevenLabs API availability â€” acceptable for POC
